下面我把这个项目**当成一个你完全没接触过的产品**来讲：它到底在干什么、有哪些部件、它们之间的关系是什么、以及“数据如何从输入一路流到输出”。我只讲整体过程与数据流，不讲代码细节。

---

## 1）用一句话理解这个项目

这是一个“离线自动控制终端”：

> 它会**看时间**（RTC）、**看温度**（温度传感器）、**听你的命令**（按键/红外/串口），然后按照你设定的**日程与规则**，去控制一个**执行器**（步进电机/蜂鸣器/LED），并把过程**显示出来**并**记录日志**。断电后还能把你之前的设置恢复出来。

你可以把它理解成一个小型“工业控制面板/智能设备控制器”，不是单纯点灯。

---

## 2）项目里有哪些“角色”（模块概念，而不是代码）

你脑子里先装下这 7 个角色：

1. **输入源**（Input Sources）

   * 温度传感器：给“温度数据”
   * RTC：给“当前时间”
   * 按键/键盘：给“用户操作”
   * 红外：给“遥控操作”
   * 串口（可选）：给“电脑命令/调试输入”

2. **系统状态**（System State）

   * 当前模式：AUTO / MANUAL / SAFE
   * 当前执行器位置：例如 0~100%
   * 当前温度、当前时间
   * 当前错误状态（有没有传感器故障等）

3. **配置数据（持久化）**（Config Stored in EEPROM）

   * 你设置过的阈值（高阈值/低阈值/冷却时间）
   * 你设置过的日程任务（每天几点做什么）
   * 你设置过的红外按键映射（某遥控键=某动作）
   * 步进电机参数（例如开度对应步数）

4. **规则与日程引擎**（Decision Makers）

   * 日程引擎：到了某个时间点就触发动作
   * 温控规则：温度超过阈值触发动作，降回阈值触发恢复动作
     它们本质是“判断器”：不断看数据，决定要不要触发。

5. **动作中心**（Action Center）

   * 所有输出都必须经过它
   * 它负责统一动作：比如 MOVE_TO(70%)、STOP、BEEP(3)
   * 它决定优先级：例如手动 STOP 必须立刻打断自动动作

6. **输出设备**（Outputs）

   * 步进电机：执行“移动到某位置”
   * 蜂鸣器：告警/提示音
   * LCD：显示状态/设置界面
   * （可选）LED/点阵/数码管：状态指示

7. **日志系统**（Observability）

   * 每一次“触发了什么、谁触发的、结果如何、何时发生”的记录
   * 主要通过串口输出，必要时也可在屏幕上简要显示

## 3）这个项目“从上电到运行”的完整故事（整体过程）

### 阶段 A：上电启动（Boot）

1. 系统上电
2. 读取 EEPROM 里的**配置**（阈值、日程、红外映射等）

   * 如果配置有效：加载成功
   * 如果配置损坏：加载默认配置，并进入 SAFE 或提示错误（看你设定）
3. 读取 RTC 当前时间作为“当前时间”
4. 初始化显示：LCD 上出现主界面（时间/温度/模式/开度）
5. 系统进入主循环：开始持续运行

> 上电阶段的关键：把“上一次保存的设置”恢复回来，这样它才像设备而不是一次性玩具。

---

### 阶段 B：持续运行（Run Loop）

从这一刻起，系统一直在做一个循环：
**采集数据 → 产生事件 → 判断是否触发 → 下发动作 → 更新显示/日志 → 回到采集**

你可以理解成一条流水线。

---

## 4）最重要：数据如何流转（按数据类型逐条讲）

### 4.1 时间数据如何流转（RTC → 日程引擎 → 动作中心 → 执行器）

1. RTC 提供当前时间（例如 12:30:00）
2. 日程引擎拿到这个时间，去对比“配置中的任务表”

   * 例如任务表里有：每天 12:30 执行 MOVE_TO(80%)
3. 如果时间匹配，就产生一个“动作请求”

   * 请求内容：`Action = MOVE_TO(80%)`
   * 请求来源：`Source = Schedule`
4. 动作中心收到请求，检查当前模式与优先级

   * 如果在 AUTO：允许执行
   * 如果在 MANUAL：通常不执行（或排队，看你定义）
   * 如果在 SAFE：拒绝执行
5. 允许执行时 → 下发到步进电机输出
6. 同时更新：

   * 系统状态（当前开度变化）
   * 日志（记录“12:30执行了什么任务”）
   * LCD（显示当前开度和最近一次动作）

**一句话总结**：时间是“触发源”，日程是“规则”，动作中心是“裁判”，执行器是“干活的”。

---

### 4.2 温度数据如何流转（DS18B20 → 温控规则 → 动作中心 → 输出）

1. 温度传感器输出温度（例如 31.2℃）
2. 系统把它写入“当前温度”状态
3. 温控规则模块拿到温度，与配置阈值比较：

   * 例如：TH=30.0℃、TL=29.0℃
4. 若温度跨过高阈值，规则产生“动作请求”：

   * `Action = MOVE_TO(100%)` 或 `BEEP(1)` 或组合
   * `Source = TempRule`
5. 动作中心进行仲裁（同上）
6. 执行器动作后：更新状态、记录日志、刷新显示

**关键理解点**：温控是“条件触发”，它不关心你怎么移动，只负责说“现在应该触发哪个动作”。

---

### 4.3 用户操作数据如何流转（按键/红外/串口 → UI/映射 → 动作中心）

用户操作有三类，它们最终都会变成**同一种东西：动作请求**。

#### ① 按键/键盘

* 你按下“模式切换”键
* UI 模块将它解释为：`Action = MODE_SET(MANUAL)`
* 投递给动作中心
* 动作中心改变系统模式，并记日志、更新显示

你按下“开到100%”键

* UI解释为：`Action = MOVE_TO(100%)`
* 动作中心下发给步进

#### ② 红外遥控

* 红外模块解码得到“键值 code”
* 去查“红外映射表（配置中保存）”：

  * 例如 code=0x45 → MOVE_TO(0%)
* 查到动作后，投递给动作中心
* 动作中心执行

#### ③ 串口命令（可选）

* 你在电脑输入 `move 70`
* CLI 将命令解析成：`Action = MOVE_TO(70%)`
* 同样投递给动作中心

**统一原则**：无论按键、红外还是串口，最后都转成动作请求，走同一条输出链路。这样系统不会乱。

---

### 4.4 配置数据如何流转（UI/串口设置 → Config → EEPROM → 下次上电恢复）

配置数据是“设备记忆”，它的流转是闭环的：

1. 你通过 UI 修改了阈值/日程
2. 系统先更新内存中的 Config
3. 你选择“保存”
4. Config 写入 EEPROM
5. 下次上电，从 EEPROM 读回来继续使用

**你要记住**：

* 配置不是每次都写（频繁写会慢、会磨损），一般是你确认保存才写。
* 系统运行时读的是“内存中的配置”，EEPROM是“备份存档”。

## 5）一个完整的“真实场景例子”（帮助你一眼看懂）

假设你设置了：

* 模式：AUTO
* 温控：T>30.0℃ → MOVE_TO(100%)，T<29.0℃ → MOVE_TO(30%)
* 日程：每天 07:30 → MOVE_TO(80%)

### 运行中会发生什么？

1. 07:29：LCD显示时间温度，电机保持当前开度
2. 07:30：RTC到点 → 日程触发 → 动作中心执行 MOVE_TO(80%)

   * LCD显示开度变为80%，日志记录“日程触发”
3. 中午温度升高到 30.2℃：温控触发 → MOVE_TO(100%)

   * 日志记录“超温触发”
4. 你突然按 STOP：按键事件 → 动作中心立即 STOP，并切入 MANUAL（如果你定义手动优先）

   * 从此 AUTO不再干预，直到你切回 AUTO
5. 若温度传感器掉线：错误管理报告故障 → SAFE模式

   * 自动控制被禁止，避免错误情况下乱控制

你看，整个系统就是一条“数据驱动动作”的流水线。

---

## 6）你需要掌握的“项目心智模型”（极简版）

如果你只能记住三句话，就记这三句：

1. **输入（时间/温度/操作） → 事件 → 决策（规则/日程） → 动作 → 输出**
2. **所有输出必须走动作中心**，不允许各模块直接抢控制权
3. **配置存在EEPROM里，运行用内存配置**，保存才写入
