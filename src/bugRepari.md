# DS1302 问题修复说明

## 1. 问题现象
- 向 DS1302 写入时间后，LCD 显示的年/月/日/时有时错位。
- 秒值可读，但日期和月份偶发错误。
- 小时在部分情况下显示异常（与 12/24 小时格式相关）。

## 2. 根因分析
- 读写时序与已验证可用版本不一致，导致位采样可能错位。
- 历史版本中存在临时代码和重复接口，维护时容易引入行为偏差。
- 小时寄存器存在 12/24 小时两种格式，未统一处理时会出现显示偏差。

## 3. 修复策略
- 保持当前项目已验证可用的读写时序，不额外改动基础波形策略。
- 所有时间字段统一执行 BCD <-> 十进制转换。
- 初始化阶段清除 CH 位（秒寄存器 bit7），保证振荡器正常运行。
- 读取小时时兼容 12/24 小时模式，统一转换成 24 小时制输出。

## 4. 当前验证流程
- 上电后写入测试时间：`2026-02-17 16:41:00`。
- 主循环调用 `DS1302_ReadTime` 读取并在 LCD 显示：
  - 第 1 行：`20YY-MM-DD`
  - 第 2 行：`HH:MM:SS`
- 观察秒递增、分钟进位、日期与月份显示是否稳定正确。

## 5. 后续建议
- 若需要量产稳定性，可加“仅首次上电写时间”的控制逻辑，避免每次复位覆盖时钟。
- 如需进一步抗干扰，可在读取时间时做两次采样一致性校验。


## 6. UART显示全为FFFF问题复盘（2026-03-05）

### 6.1 问题现象
- Windows 端日志显示 `T->S` 数据正常，例如：`55 AA 01 03 1D 01 1E 40`。
- MCU 侧 LCD1602 显示基本全为 `FF`，偶发解码异常。

### 6.2 根因
- RX/TX 缓冲使用了 `xdata`，在无可用 XRAM 的目标板上，读写值会漂到 `0xFF`。
- 显示缓存也放在 `xdata`，放大了“全 FF”现象。
- 串口中断接收顺序“先清 RI 再读 SBUF”在边界时序下不稳。

### 6.3 修复
- `bsp_uart.c`：`s_rx_buf/s_tx_buf` 从 `xdata` 改为 `idata`。
- `bsp_uart.h`：串口缓冲从 64 下调到 32，降低内部 RAM 压力。
- `main.c`：显示缓存放入内部 RAM，避免大块 `xdata` 依赖。
- `Uart_ISR`：改为先读 `SBUF`，再清 `RI`。

### 6.4 结果
- LCD 能正常显示数据，不再出现满屏 `FF`。
- 后续方案：在 `main.c` 按协议帧做命令翻译并显示到 LCD。
